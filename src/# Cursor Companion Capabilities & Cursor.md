# Cursor Companion Capabilities & Cursor Integration

**Cursor Companion** is a lightweight, native macOS menu bar application (SwiftUI + AppKit) designed to monitor personal usage of the **Cursor IDE**. It allows developers to check token consumption, models used, and spending without needing to open the Cursor dashboard in a web browser.

---

## 1. Zero-Config Local Authentication

The app features a zero-friction auth model, reading your existing signed-in Cursor session from your local machine.

* **Local Database Access:** The app queries the SQLite database used by Cursor to store application state:
  * **Path:** `~/Library/Application Support/Cursor/User/globalStorage/state.vscdb`
  * **Access Method:** Opened strictly in **read-only + immutable** mode to prevent database locking or corruption.
* **Credentials Read:**
  * `cursorAuth/accessToken` (JWT)
  * `cursorAuth/cachedEmail` (for display in UI header)
  * `cursorAuth/stripeMembershipType` (for plan recognition)
* **Cookie Formulation:** The app reads the JWT and dynamically builds a session cookie (`WorkosCursorSessionToken`) in the format `<userId>::<jwt>`, extracting `userId` from the `sub` claim of the JWT.
* **Security:**
  * Credentials and tokens are held **only in volatile memory** during execution and refresh cycles.
  * The app **never persists** any tokens or credentials to its own storage, UserDefaults, plists, or logs.

---

## 2. API Integration with Cursor

Cursor Companion queries unofficial, reverse-engineered `cursor.com` endpoints using the formulated session cookie:

* **Base URL:** `https://cursor.com`
* **Session Cookie:** `Cookie: WorkosCursorSessionToken=<userId>::<jwt>`
* **HTTP Headers:** Includes `Origin: https://cursor.com` and `Referer: https://cursor.com/dashboard/usage` to match expected dashboard request patterns.

### Endpoints Queried:
1. **`GET /api/usage-summary`:**
   * Used to retrieve general plan parameters (membership type, billing cycle dates, pooled team/individual quotas, and current billing cycle on-demand spending).
2. **`POST /api/dashboard/get-filtered-usage-events`:**
   * Retrieves individual usage events (model, timestamp, token counts, and cost details).
   * Request specifies `startDate` and `endDate` range bounds as epoch millisecond strings.

---

## 3. What the App Tracks & Displays

From the retrieved API data, Cursor Companion monitors and aggregates the following metrics:

* **Token Consumption:** Tracks details of all token-based requests:
  * **Input Tokens:** Code/prompts sent to the model.
  * **Output Tokens:** Completions generated by the model.
  * **Cache Write vs. Cache Read Tokens:** Identifies how much cache reuse is occurring (cache reads are often highly discounted).
* **Model Usage Patterns:** Analyzes which AI models are being used (e.g., Claude, GPT) and provides:
  * Number of requests per model.
  * Total token split per model.
  * Spend incurred per model.
* **Cost & Spend Tracking:**
  * **Authoritative Spend:** Sums the `chargedCents` field from events (which includes model cost + Cursor token fees).
  * **On-Demand Costs:** Tracks total on-demand spending in the current billing cycle.
  * **Projection & Burn Rate:** Estimates end-of-cycle spend based on current usage trends.
* **Activity Metrics:**
  * Total requests.
  * Headless / background-agent requests.
  * Busiest day.

---

## 4. Passive UI & Local Caching

* **Menu Bar UI:** The app resides entirely in the macOS system status menu bar. It features a compact overview indicator and opens a details panel when clicked.
* **Caching:** App caches up to ~90 days of `UsageEvent`s and the latest summary snapshots locally in `~/Library/Application Support/CursorCompanion` to allow offline viewing and minimize API requests.
* **Refresh Strategy:** Automatically fetches new data incrementally (looking only for events after the last known timestamp) every 60 minutes (configurable), or on manual trigger.
